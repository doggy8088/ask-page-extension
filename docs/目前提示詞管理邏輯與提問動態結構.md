# 目前提示詞管理邏輯與提問動態結構

## 文件目的
本文件整理 AskPage 目前「提示詞」相關的管理邏輯，聚焦三件事：**哪些內容算 prompt、哪些 prompt 可以被使用者管理、實際送出提問時 prompt 是怎麼被動態組裝的**。如果要看頁面文字 / HTML 本身如何擷取，可再搭配閱讀 `docs/目前獲取頁面內容的演算法分析.md`。

## 一、目前生效的提示詞管理入口
目前實際生效的管理入口是 **`settings.html` / `settings.js` 的選項頁**，因為 `manifest.json` 只宣告了 `options_ui`，沒有設定 `default_popup`；也就是說，現在的 prompt 管理主流程是走選項頁，不是 popup（`manifest.json:17-20`; `settings.js:74-85`）。

這個選項頁目前真正可編輯的是兩類內容，但整個 prompt 系統實際上包含下列幾類資料：

| 類型 | 儲存鍵 | 是否可編輯 | 說明 |
| --- | --- | --- | --- |
| 內建摘要提示詞 | `CUSTOM_SUMMARY_PROMPT` | 是，但只限 `/summary` | `/summary` 的實際 prompt 內容 |
| 自訂斜線命令 | `CUSTOM_COMMANDS` | 是 | 每筆包含 `cmd` 與 `prompt` |
| 自訂系統提示詞 | `CUSTOM_SYSTEM_PROMPT` | 是 | 附加在程式產生的固定系統提示詞之後 |
| 固定系統提示詞 | 無獨立儲存鍵 | 否 | 由 `content.js` 執行時依模式動態產生 |
| 頁面上下文 | 無獨立儲存鍵 | 否 | 由頁面內容、選取文字、模式開關動態產生 |
| 對話歷史 | 記憶體內 `conversationHistory` | 否 | 追問時自動帶入 |
| 提問歷史 | `ASKPAGE_PROMPT_HISTORY` | 否 | 只供輸入框上下鍵瀏覽，不是 provider 的永久設定 |

## 二、目前有哪些「提示詞」來源

### 1. 使用者直接輸入的問題
一般提問時，輸入框的文字會先經過 `trim()`，然後直接成為 `question`（`content.js:1020-1024`）。這是最直接的 prompt 來源。

### 2. 內建 `/summary`
`/summary` 不是獨立模式，而是先把命令改寫成一段真正送給模型的 prompt。邏輯如下（`content.js:1038-1043`）：

1. 先讀 `CUSTOM_SUMMARY_PROMPT`
2. 如果有值，就用使用者自訂內容
3. 如果沒值，就退回預設摘要 prompt

目前預設摘要 prompt 是：

```text
請幫我總結這篇文章，並以 Markdown 格式輸出，內容包含「標題」、「重點摘要」、「總結」
```

這段預設值目前是由 `content.js` 在執行 `/summary` 時直接作為 fallback 使用（`content.js:1038-1043`）。

### 3. 自訂斜線命令
如果輸入以 `/` 開頭，系統會先到 `CUSTOM_COMMANDS` 中找同名命令；若找到，就把 `question` 改寫成該命令的 `prompt`，再走一般 AI 提問流程（`content.js:1063-1073`）。

也就是說：

- UI 上看到的是命令名稱，例如 `/translate`
- 實際送給模型的是命令對應的 `prompt`

若找不到對應命令，則只回覆「未知命令」，**不會送到 AI provider**（`content.js:1073-1079`）。

### 4. 系統提示詞
除了使用者問題本身，還有一層由程式生成的 `systemPrompt`。`buildSystemPrompt()` 會依頁面上下文格式與選取狀態動態組裝固定規則，再把設定頁的 `CUSTOM_SYSTEM_PROMPT` 附加在後方。詢問模式一旦建立對話快照，這份結果會凍結到 `/clear` 或頁面重新整理；中途修改設定只會套用到下一個新對話。

它的固定骨架如下：

```text
You are a helpful assistant that answers questions about web page content.
{pageContextDescription}
{selectedTextDescription}
{screenshotDescription}
Think before acting: state assumptions explicitly, surface ambiguity, and ask a concise clarifying question instead of guessing when the request is unclear or has multiple valid interpretations.
Prefer the simplest solution that fully satisfies the request. Avoid speculative features, unnecessary abstractions, extra configurability, and impossible-scenario handling.
Make surgical changes only. Do not refactor or improve unrelated code, formatting, or comments. Clean up only what your own changes make obsolete.
For non-trivial tasks, define a short plan and clear success criteria before acting, then verify that each meaningful step moved you toward that goal.
If there is a simpler or safer approach than the user implied, say so briefly and prefer it unless the user clearly asked otherwise.
You are in agent mode when page context is HTML. Use the available page tools whenever the user asks you to inspect or modify the current page, selected text, or form fields. In particular, you can use run_js to read or modify the current page DOM, inline styles, classes, attributes, text, layout, and behavior.
Do not say that you cannot directly modify the page, HTML, DOM, or CSS when the change can be done through the available tools. Prefer performing the change with tools instead of refusing for capability reasons.
Never claim that a page change succeeded unless the corresponding tool result confirms it.
For non-trivial form filling, inspect the form fields first before mutating them.
Please format your answer using Markdown when appropriate.
As a default, provide responses in zh-tw unless specified otherwise.
Do not provide any additional explanations or disclaimers unless explicitly asked.
No prefix or suffix is needed for the response.
```

其中三個動態區塊會依情況改變，另外固定規則也會要求模型落實：

- 先說明假設、遇到歧義時先釐清，不要默默猜測
- 優先採用最簡單、最小變更的做法
- 不要擴大修改範圍到無關程式碼
- 多步驟任務先定義簡短計畫與成功條件，再逐步驗證

而三個動態區塊則會依情況改變：

- `{pageContextDescription}`：說明目前上下文是純文字頁面內容，還是過濾後的 HTML。
- `{selectedTextDescription}`：有選字時會強調「選取文字是主要焦點」；沒有選字時則改成「整頁內容是主要參考」。
- `{screenshotDescription}`：建立最新頁面上下文時若本輪包含截圖才會加入；詢問模式為維持固定前綴，不會因後續圖片附件重建系統提示詞。

詢問模式的固定規則也會把頁面上下文與選取文字標記為不受信任資料，要求模型只能將其作為分析來源，不得遵循頁面內容中的指令、規則變更要求或工具操作要求。

### 5. 頁面上下文 prompt
真正送到模型前，還會建立一段 `conversationContextText`，由 `buildConversationContextText()` 產生（`content.js:426-445`）。它不是 system prompt，但屬於 prompt 結構的一部分。

它的內容固定由兩層來源組成：

1. **整頁內容**
2. **選取文字（如果存在）**

組裝規則如下：

- 詢問模式：標記為 DOM 推導的近似 Accessibility Tree，只做內容問答
- 代理模式：標記為 `Filtered page HTML context...`，允許工具調用
- 有選字時，再追加 `Selected text (main focus):`

### 6. 對話歷史 prompt
多輪對話時，所有可送入模型的前文都會按原始順序帶回。程式不會自動裁切、摘要或壓縮對話歷史。

- Gemini：把歷史保留成獨立的 `user` / `model` contents
- OpenAI / Azure OpenAI / OpenAI Compatible：把歷史保留成 message array 或等價的 Responses API items
- Anthropic：把歷史保留成 `user` / `assistant` messages

各使用者回合的截圖與手動附加圖片會保留在原回合，不會只附在最新問題上。進度、推理摘要與工具 trace 雖然保留於對話 UI，仍以 `includeInModelContext: false` 排除。關鍵點是：**歷史中保存的是實際 prompt 內容，不是斜線命令字面值。**

## 三、提示詞管理的資料結構

### 1. 內建命令
目前內建命令有四個（`settings.js:78-84`; `content.js:771-776`）：

| 命令 | 是否送 AI | 是否可編輯 | 用途 |
| --- | --- | --- | --- |
| `/clear` | 否 | 否 | 清除提問歷史與對話狀態 |
| `/summary` | 是 | 是 | 送出摘要 prompt |
| `/screenshot` | 否 | 否 | 切換截圖模式 |
| `/agent` | 否 | 否 | 切換詢問模式 / 代理模式 |

其中只有 `/summary` 被標記為 `editable: true`，而且它不是寫回 `CUSTOM_COMMANDS`，而是單獨存到 `CUSTOM_SUMMARY_PROMPT`（`settings.js:487-492`）。

### 2. 自訂命令
自訂命令存放在 `CUSTOM_COMMANDS`，資料格式是陣列，每筆大致如下（`settings.js:500-531`; `content.js:1065-1070`）：

```js
{
    cmd: '/example',
    prompt: '這是一段真正會送給 AI 的提示內容'
}
```

命令名稱規則如下（`settings.js:404-457`）：

- 必須以 `/` 開頭
- 只能包含英數字、底線、連字號
- 不能與內建命令重複
- 不能與現有自訂命令重複

### 3. 兩套不同的「歷史」
目前實作裡有兩套容易混淆的歷史：

| 名稱 | 儲存位置 | 作用 | 上限 |
| --- | --- | --- | --- |
| `conversationHistory` | 記憶體 | 讓模型保留前文脈絡 | 不自動裁切；到 `/clear` 或重新整理頁面為止 |
| `ASKPAGE_PROMPT_HISTORY` | `chrome.storage.local` | 讓輸入框可用上下鍵回看之前輸入 | 100 筆（`content.js:81`; `content.js:916-917`; `content.js:1083-1086`） |

其中 `ASKPAGE_PROMPT_HISTORY` 儲存的是**改寫後的真實 prompt**。例如：

- 輸入 `/summary`，歷史裡存的是摘要 prompt 文字，不是 `/summary`
- 輸入自訂命令 `/translate`，歷史裡存的是 `/translate` 對應的 `prompt`

這是因為 `promptHistory.push(question)` 發生在命令展開之後（`content.js:1038-1043`; `content.js:1068-1086`）。

## 四、提問時的動態組裝流程

### 1. 對話框開啟時先捕捉一次選取文字
`createDialog()` 在開啟對話框當下就把 `window.getSelection().toString().trim()` 存進 `capturedSelectedText`（`content.js:510-515`）。這份選字會作為本輪與後續追問的焦點來源。

### 2. `handleAsk()` 先把輸入轉成真正的 `question`
`handleAsk()` 會依序判斷（`content.js:1020-1092`）：

1. 空字串就不處理
2. `/clear`：清掉 prompt history 與 conversation history
3. `/summary`：改寫成摘要 prompt
4. `/screenshot` / `/agent`：切換模式，不送 AI
5. 自訂命令：改寫成對應 `prompt`
6. 一般提問：直接使用原始文字

這裡同時有兩個欄位：

- `question`：真正送給 AI 的內容
- `displayedQuestion`：畫面上顯示給使用者看的內容

因此像 `/summary` 或自訂命令這種情境，UI 可以顯示命令名稱，但模型實際看到的是展開後的 prompt（`content.js:1022-1023`; `content.js:1041-1043`; `content.js:1069-1071`; `content.js:1502-1503`）。

### 3. 建立或重用頁面對話上下文
接著 `getPageConversationContext()` 會依模式建立或重用由 `preparePageConversationContext()` 產生的三個核心欄位：

- `pageContext`
- `systemPrompt`
- `conversationContextText`

其中 `pageContext` 又取決於兩個模式：

| 模式 | 內容來源 | 重要行為 |
| --- | --- | --- |
| 詢問模式 | 從 `document.body`、DOM 語意與 ARIA 屬性推導的近似 Accessibility Tree | `/clear` 後第一問建立快照，後續不重新讀取 DOM；不啟用工具調用 |
| 代理模式 | 過濾後的 `outerHTML` | 即使有選字也保留完整過濾後 HTML，並允許工具調用；選字只作為目標區域焦點 |

詢問快照包含頁面上下文、選取文字、系統提示詞與自訂系統提示詞。關閉再開啟對話框、切換 provider 或暫時使用代理模式都不會重置；`/clear` 與頁面重新整理才會開始新的詢問快照。

### 4. 選取文字會持續影響後續追問
回答完成後，系統會把本輪的 `capturedSelectedText` 存進 `conversationSelectedText`（`content.js:1501`; `content.js:1638`; `content.js:1797`; `content.js:1876`）。下一輪若沒有重新選字，`getActiveSelectedText()` 會沿用這份舊 selection（`content.js:400-402`）。

這代表 AskPage 的 prompt 焦點是有狀態的；只有 `/clear` 會把這份狀態一起清掉（`content.js:498-500`; `content.js:1026-1035`）。

## 五、各 provider 實際送出的 prompt 結構

### 1. Gemini
Gemini 詢問模式的 request body 核心結構如下：

```js
{
    systemInstruction: {
        parts: [{ text: systemPrompt + '\n\n' + conversationContextText }]
    },
    contents: [
        { role: 'user', parts: [{ text: firstQuestion }] },
        { role: 'model', parts: [{ text: firstAnswer }] },
        { role: 'user', parts: [{ text: followUpQuestion }] }
    ],
    generationConfig: {
        temperature: 0.7,
        topP: 0.95,
        maxOutputTokens
    }
}
```

Gemini 有兩個特點：

1. 詢問快照位於固定的 `systemInstruction` 前綴。
2. 歷史使用原生多輪角色，不再重建 transcript，也不會重複加入本輪問題；圖片以 `inline_data` 留在原始使用者回合。

### 2. OpenAI / Azure OpenAI / OpenAI Compatible
這三條路徑仍共用同一組對話上下文與工具迴圈，但 **gpt-5\*** 會改走 Responses API，其他模型維持 chat/completions 風格。每次請求都從完整 `conversationHistory` 展開訊息，本輪問題不會再由 request builder 重複附加。

非 gpt-5 模型的核心結構是一致的：

```js
[
    {
        role: 'system',
        content: `${systemPrompt}\n\n${conversationContextText}`
    },
    ...conversationHistory
]
```

差異主要只在 API URL 與 token 參數：

| Provider | 額外差異 |
| --- | --- |
| OpenAI | 非 gpt-5 模型使用 `chat/completions`；`gpt-5*` 強制改用 `/v1/responses`。詢問模式會附加同一對話固定的 `prompt_cache_key`。 |
| Azure OpenAI | 非 gpt-5 deployment 使用 deployment-based `chat/completions`；`gpt-5*` deployment 強制改用 `/openai/v1/responses`，並在需要時自動改用可支援的 API version。 |
| OpenAI Compatible | 一般情況維持 `chat/completions`；若模型名稱是 `gpt-5*`，也會改走 `responses` 風格請求。 |

Azure OpenAI 與 OpenAI Compatible 不會被加入 OpenAI 官方端點專屬的 `prompt_cache_key`。Anthropic 詢問模式另外使用頂層 `cache_control: { type: 'ephemeral' }`。所有支援的 provider 都會把截圖與手動附加圖片保留在原始使用者回合中。

### 3. Context Window 超限
程式不會對詢問模式的語意樹、fallback 純文字、選取文字或對話歷史設定字元上限，也不會預先裁切或估算後自動改寫。若完整快照與所有對話超過模型 Context Window，供應商 API 的錯誤會原樣進入既有錯誤處理流程。**只有使用者可以執行 `/clear` 開始新對話；程式不會代替使用者自動清除。**

## 六、哪些內容「可管理」，哪些內容「不可管理」

### 可由使用者管理
- `/summary` 的實際摘要 prompt
- 自訂斜線命令的名稱與 prompt 內容
- 附加在固定規則後方的自訂系統提示詞

### 不可由使用者直接管理
- `buildSystemPrompt()` 的固定規則
  - 現在更明確要求模型只做最小必要規劃，盡快產生可見結果，避免把輸出額度耗在冗長規劃上
  - 在代理模式下會明確要求模型使用頁面工具，尤其是 `run_js`，直接修改目前頁面的 DOM / CSS，而不是聲稱自己不能改頁面
- `conversationContextText` 的段落結構
- 對話歷史是否帶回、帶回格式
- provider request body 的欄位配置

換句話說，**使用者可管理命令 prompt 與自訂系統提示詞，但不可改變程式建立的固定系統層結構、快照生命週期與 provider request 格式。**

## 七、維護時最容易忽略的幾個重點

1. `/summary` 與自訂命令都只是「先展開成 prompt，再走一般提問流程」，不是獨立 provider 流程。
2. UI 顯示的命令名稱，不等於真正送給模型的 prompt；模型看到的是展開後內容。
3. prompt history 與 conversation history 是兩套不同機制，不能混為一談。
4. 詢問模式的 DOM 語意快照只在第一問建立，後續頁面變動不會自動更新。
5. 對話歷史不自動裁切；Context Window 超限時必須由使用者執行 `/clear`。
6. `conversationSelectedText` 會讓選取文字跨輪延續，而詢問快照建立後會固定該次選取焦點。

## 九、GPT-5 (Responses API) 的相容性與降級機制

### 1. 格式不相容背景
對於 `gpt-5*` 系列模型，本專案預設會使用 OpenAI 的 Responses API 格式（如 `/v1/responses` 或者是 Azure 相關的 responses 端點）。Responses API 返回的 JSON 結構以 `output` 陣列為主，且 SSE 串流事件類型（例如 `response.output_text.delta`）與傳統 `chat/completions`（通常是 choice-based 結構，事件名稱通常不具備特定 Responses type）完全不同。

然而，當使用者透過第三方中轉服務、自建反代、甚至是本地推論代理（如 OneAPI、NewAPI、Ollama 轉接等）使用 `gpt-5*` 模型時，即便端點是 `/responses`，後端仍可能回傳傳統的 `chat/completions` 格式。

### 2. 系統採用的向下相容與防錯策略
為了解決不相容導致解析空文字的錯誤（`OpenAI 已完成回應，但內容不是可顯示的文字`），系統在 `content.js` 中實作了以下容錯與降級機制：

- **自動識別與降級 (Fallback)**：
  - **非串流模式**：在 `normalizeResponsesApiResponse` 中，若發現 `responseData` 具有 `choices` 陣列，系統會自動將其識別為傳統 `chat/completions` 格式並直接返回，不進行 Responses API 的轉換。
  - **串流模式**：在 `fetchResponsesApiStream` 串流中，若在 `onEvent` 收到含有 `choices` 欄位或 `object === 'chat.completion.chunk'` 的事件，系統會在記憶體中切換為 `chat/completions` 的串流解析邏輯，在串流結束時重組為相容格式返回。

- **多事件 Text Delta 寬容解析**：
  - 為相容不同中轉代理或橋接器，在 Responses 串流中，系統同時監聽 `response.output_text.delta`、`response.text.delta`、`response.content_part.delta` 以及 `response.output.delta`。
  - 文字提取過程具備高容錯性，會自動嘗試取得 `payload.delta`、`payload.delta.text` 與 `payload.text`。

- **明確的錯誤訊息拋出**：
  - 許多代理在遇到額度不足、超時、或後端故障時會回傳 `200 OK` 搭配帶有 `error` 的 JSON 物件。系統現在會在非串流與串流模式的最初解析點攔截此 `error` 欄位並立即拋出具體錯誤（例如 `API 回報錯誤：...`），避免該錯誤 JSON 被誤判為空內容回应。
