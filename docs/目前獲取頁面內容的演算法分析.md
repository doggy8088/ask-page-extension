# 目前獲取頁面內容的演算法分析

## 文件目的
本文件說明 AskPage 目前如何在使用者開啟對話框、輸入問題、送出至各 AI provider 之前，擷取與組合頁面內容。重點不是 UI 操作本身，而是「哪些文字會被拿來當作上下文、什麼時候抓、如何保留、在哪裡截斷」，方便後續維護者評估功能調整是否會影響問答品質與使用者預期。

## 一、整體流程
整體資料流可以拆成四段：**觸發對話框 → 捕捉初始 selection → 送出前重抓整頁文字 → 依 provider 組成請求**。

首先，背景腳本負責接住鍵盤快捷鍵與點擊擴充功能圖示兩種入口。當收到 `toggle-dialog` 指令時，`background.js` 會將訊息送到當前頁面的 content script，要求切換提問視窗（`background.js:1-46`, `background.js:90-129`）。這表示頁面內容的真正擷取不在背景腳本，而是在 content script 內發生。

接著，content script 在收到 `toggle-dialog` 後呼叫 `createDialog()` 建立對話框。如果這次是「開啟」而不是「關閉」，它會立刻呼叫 `window.getSelection()`，並把當下瀏覽器實際反白的文字轉成字串後 `trim()`，存進區域變數 `capturedSelectedText`（`content.js:24-60`, `content.js:377-382`）。這一步非常重要：**選取文字只在對話框開啟當下抓一次，不會在每次送出問題前重新讀取瀏覽器 selection。**

當使用者在對話框輸入問題並送出後，`handleAsk()` 會判斷這次輸入是一般提問、內建斜線命令還是自訂命令。如果是一般提問或 `/summary`，最終都會落到 `askAI(question, getActiveSelectedText(capturedSelectedText), displayedQuestion)`，也就是把目前有效的選取文字交給 provider 層處理（`content.js:549-645`）。

最後，真正送 API 之前，各 provider 實作都會重新執行 `getFullPageText()` 取得整頁文字，再將「整頁文字、選取文字、對話歷史、目前問題」依各自 API 格式組成請求內容。Gemini、OpenAI、Azure OpenAI、OpenAI Compatible 四條路徑都如此（`content.js:894-1057`, `content.js:1061-1194`, `content.js:1198-1354`, `content.js:1356-1436`）。也就是說，**選取文字偏向一次捕捉、跨輪沿用；整頁文字則是每次提問都重新抓。**

## 二、預設選取範圍
目前實作中的「預設選取範圍」非常單純，就是瀏覽器原生 selection。程式只在 `createDialog()` 內呼叫一次 `window.getSelection()`，然後直接取 `toString().trim()` 作為選取內容（`content.js:380-381`）。Repo 內與 selection 相關的實作只有這一處，因此可以明確判斷：

1. 系統**不會**自動向前後段落擴張上下文。
2. 系統**不會**推測目前游標所在段落或 DOM 區塊。
3. 系統**不會**根據標題、段落、節點層級自動補足 surrounding context。

換句話說，若使用者只反白一句話，送進去的「選取重點」就只有那一句；真正補足上下文的來源不是 selection 自身擴張，而是後續額外附帶的整頁文字。

另一個容易忽略的點是，這份 selection 是在對話框開啟當下取得。如果使用者開啟對話框後才回到頁面重新反白其他文字，這次新的選取不會自動覆蓋 `capturedSelectedText`，除非重新關閉再開啟對話框，或者依賴後面提到的 `conversationSelectedText` 保留機制。

## 三、選取內容如何與上下文組合
頁面文字的主來源是 `getFullPageText()`。它先用 `getPageContextContainer()` 選擇主要容器：優先取 `main`，若頁面沒有 `main`，再退而求其次選「只有一個 `article`」的情況；若兩者都不符合，最後才退回 `document.body`（`content.js:298-309`）。選到容器後，直接讀 `.innerText` 並截到前 15000 字元（`content.js:311-313`）。因此目前演算法偏向「抓可見文字的主內容區」，而不是直接序列化完整 DOM。

選取文字與整頁文字的組合則由 `buildConversationContextText()` 處理。如果存在選取文字，格式固定是：先放 **Full page content**，再放 **Selected text (main focus)**，且 selected text 只保留前 5000 字元（`content.js:335-341`）。這代表系統的策略不是「只看選取內容」，而是「整頁為背景、選取為焦點」。

實際送出時，Gemini 路徑會把這段文字 context 再接上對話歷史 transcript，以及 `Current question`，組成一個主要文字 part（`content.js:343-353`, `content.js:938-941`）。如果截圖功能啟用，Gemini 只是在文字 part 之外再附加 screenshot part；文字頁面內容仍然來自 `getFullPageText()`，沒有改變文字擷取演算法（`content.js:921-955`）。

OpenAI、Azure OpenAI、OpenAI Compatible 三條路徑則把 `buildSystemPrompt(...)` 與 `buildConversationContextText(...)` 放在 system message，接著串上已保存的 `conversationHistory`，最後再加入目前的 user question（`content.js:1091-1098`, `content.js:1242-1249`, `content.js:1377-1384`）。雖然封裝格式不同，但核心內容一致：**整頁文字永遠是基底，selection 是額外標示出的主要焦點。**

## 四、一般提問與 `/summary` 的差異
`handleAsk()` 對一般提問與 `/summary` 的差異，其實只在「問題文字怎麼決定」，不在「頁面內容怎麼擷取」。

一般提問時，輸入框內容會直接作為 `question` 傳進 `askAI()`（`content.js:551-553`, `content.js:642-645`）。而當輸入為 `/summary` 時，程式會先嘗試讀取自訂摘要 prompt；若沒有設定，就改寫成預設摘要提示詞，例如要求輸出標題、重點摘要與總結（`content.js:567-572`）。但之後仍然是同一條呼叫路徑：`askAI(question, getActiveSelectedText(capturedSelectedText), displayedQuestion)`（`content.js:645`, `content.js:1439-1452`）。

這表示 `/summary` 並沒有獨立的頁面擷取邏輯，也沒有保證一定只總結全頁。若目前有效 selection 存在，`buildSystemPrompt()` 仍會告訴模型「使用者選取了要優先聚焦的文字」，`buildConversationContextText()` 也仍會把該 selection 放進 `Selected text (main focus)` 區段（`content.js:319-333`, `content.js:335-341`）。因此實作上雖然 UI 文案把 `/summary` 描述成「總結整個頁面」，但它其實更接近「以目前頁面為背景，必要時優先根據選取內容做摘要」。對維護者來說，這是一個重要的產品語意落差。

另外，`/screenshot` 是獨立的設定／測試命令，只切換截圖開關與測試截圖，不走上述文字問答流程，也不會進入 `askAI()` 的一般頁面文字組裝邏輯（`content.js:574-615`）。

## 五、重要函式、資料流與長度限制
### 1. 關鍵函式
- `createDialog()`：在視窗開啟瞬間擷取 selection（`content.js:377-382`）。
- `getPageContextContainer()`：決定整頁文字從 `main`、單一 `article` 或 `body` 取哪個容器（`content.js:298-309`）。
- `getFullPageText()`：讀取容器 `.innerText` 並限制在 15000 字元內（`content.js:311-313`）。
- `getActiveSelectedText()`：優先使用本次 `capturedSelectedText`，否則 fallback 到 `conversationSelectedText`（`content.js:315-317`）。
- `buildSystemPrompt()`：決定模型是否被告知「有選取文字要優先聚焦」以及是否包含 screenshot 情境（`content.js:319-333`）。
- `buildConversationContextText()`：組裝 full page 與 selected text 的最終文字上下文（`content.js:335-341`）。
- `buildConversationHistoryTranscript()`：Gemini 專用，把既有對話歷史串成 transcript（`content.js:343-353`）。
- `handleAsk()`：將一般提問、`/summary`、`/clear`、`/screenshot`、自訂命令導向正確流程（`content.js:549-645`）。
- `askAI()`：依 provider 分流到 Gemini / OpenAI / Azure / OpenAI Compatible（`content.js:1439-1452`）。

### 2. selection 的保存與清除
`conversationSelectedText` 是跨輪對話沿用 selection 的核心狀態。若本輪送出時有 `capturedSelectedText`，在模型回答完成後會把它寫回 `conversationSelectedText`，讓下一輪即使沒有重新反白，也可持續沿用相同焦點（`content.js:1054`, `content.js:1191`, `content.js:1350`, `content.js:1428`）。

這個機制讓使用者可以先選取一段文字，再連續追問多輪，而不需要每次都重新選取。不過它也帶來狀態延續的副作用：若使用者忘記先前已有選取焦點，後續問題可能仍持續受到舊 selection 影響。

`/clear` 會呼叫 `clearConversationHistory()`，同時清掉 `conversationHistory` 與 `conversationSelectedText`（`content.js:369-372`, `content.js:555-564`）。因此 `/clear` 不只是清聊天紀錄，也是在語意上重置「目前聚焦文字」。

### 3. 長度限制
目前和頁面內容擷取直接相關的長度限制如下：

- **整頁文字上限：15000 字元**，定義在 `getFullPageText()`（`content.js:311-313`）。
- **選取文字上限：5000 字元**，定義在 `buildConversationContextText()` 中的 `capturedSelectedText.slice(0, 5000)`（`content.js:335-338`）。
- **對話歷史上限：20 則 message**，由 `MAX_CONVERSATION_MESSAGES = 20` 與 `addConversationTurn()` 維持（`content.js:13`, `content.js:362-367`）。
- **提示歷史上限：100 筆**，由 `handleAsk()` 在寫入 `PROMPT_HISTORY_STORAGE` 前裁切（`content.js:637-640`）。
- **Gemini 回覆 token 上限：2048**，在 `generationConfig.maxOutputTokens` 指定（`content.js:989-991`）。其他 provider 也各自設為約 2048 輸出上限（`content.js:1119-1123`, `content.js:1267-1271`）。

其中與「頁面內容演算法」最直接相關的是前兩項：15000 字元的 full page 與 5000 字元的 selected text。這兩個限制共同決定了模型能看到多少來源內容。

## 六、維護者應特別注意的實作含義
第一，這套演算法的核心假設是：**selection 提供焦點，full page 提供背景**。因此若未來要改善摘要或問答品質，調整點通常會落在容器選擇策略、截斷長度、或 selection 與 full page 的排序與權重，而不是單純改 prompt。

第二，`/summary` 目前不是獨立摘要模式，而是「換了一段 prompt 的一般提問」。如果產品預期 `/summary` 永遠總結全頁，則現況與產品語意並不完全一致，因為有效 selection 仍會改變模型焦點。

第三，selection 與 full page 的擷取時機不同：selection 在開框時抓一次，full page 在送出前每次重抓。這讓頁面內容若在對話過程中更新，full page 會反映最新文字；但 selection 未必同步更新。若日後遇到使用者反映「我重新選字了但沒生效」，很可能就是這個設計造成的。
