# 目前獲取頁面內容的演算法分析

## 文件目的
本文件說明 AskPage 目前如何在使用者開啟對話框、輸入問題、送出至各 AI provider 之前，擷取與組合頁面內容。重點不是 UI 操作本身，而是「哪些文字會被拿來當作上下文、什麼時候抓、如何保留、是否裁切」，方便後續維護者評估功能調整是否會影響問答品質與使用者預期。

## 一、整體流程
整體資料流可以拆成四段：**觸發對話框 → 捕捉初始 selection → 依模式取得頁面上下文 → 依 provider 組成可延續的多輪請求**。

首先，背景腳本負責接住鍵盤快捷鍵與點擊擴充功能圖示兩種入口。當收到 `toggle-dialog` 指令時，`background.js` 會將訊息送到當前頁面的 content script，要求切換提問視窗（`background.js:1-46`, `background.js:90-129`）。這表示頁面內容的真正擷取不在背景腳本，而是在 content script 內發生。

接著，content script 在收到 `toggle-dialog` 後呼叫 `createDialog()` 建立對話框。如果這次是「開啟」而不是「關閉」，它會立刻呼叫 `window.getSelection()`，並把當下瀏覽器實際反白的文字轉成字串後 `trim()`，存進區域變數 `capturedSelectedText`（`content.js:24-60`, `content.js:377-382`）。這一步非常重要：**選取文字只在對話框開啟當下抓一次，不會在每次送出問題前重新讀取瀏覽器 selection。**

當使用者在對話框輸入問題並送出後，`handleAsk()` 會判斷這次輸入是一般提問、內建斜線命令還是自訂命令。如果是一般提問或 `/summary`，最終都會落到 `askAI(question, getActiveSelectedText(capturedSelectedText), displayedQuestion)`，也就是把目前有效的選取文字交給 provider 層處理（`content.js:549-645`）。

最後，真正送 API 之前會依模式取得頁面上下文。詢問模式在 `/clear` 後的第一次提問才會從 DOM 建立近似 Accessibility Tree，並將頁面內容、選取文字、系統提示詞與自訂系統提示詞凍結成對話快照；後續追問直接重用該快照。代理模式則仍會在每次提問時取得最新的過濾後頁面 HTML。**詢問模式追求固定提示詞前綴；代理模式追求即時頁面狀態。**

重新整理頁面會重新載入 content script，因此記憶體中的詢問快照與多輪對話一併消失，效果等同開始新對話。關閉再開啟 AskPage 對話框、切換 provider，或暫時切到代理模式，都不會清除既有詢問快照；只有使用者自行執行 `/clear` 或重新整理頁面才會重置。

## 二、預設選取範圍
目前實作中的「預設選取範圍」非常單純，就是瀏覽器原生 selection。程式只在 `createDialog()` 內呼叫一次 `window.getSelection()`，然後直接取 `toString().trim()` 作為選取內容（`content.js:380-381`）。Repo 內與 selection 相關的實作只有這一處，因此可以明確判斷：

1. 系統**不會**自動向前後段落擴張上下文。
2. 系統**不會**推測目前游標所在段落或 DOM 區塊。
3. 系統**不會**根據標題、段落、節點層級自動補足 surrounding context。

換句話說，若使用者只反白一句話，送進去的「選取重點」就只有那一句；真正補足上下文的來源不是 selection 自身擴張，而是後續額外附帶的整頁文字。

另一個容易忽略的點是，這份 selection 是在對話框開啟當下取得。如果使用者開啟對話框後才回到頁面重新反白其他文字，這次新的選取不會自動覆蓋 `capturedSelectedText`，除非重新關閉再開啟對話框，或者依賴後面提到的 `conversationSelectedText` 保留機制。

## 三、選取內容如何與上下文組合
頁面上下文的主來源是 `getPageContext()`。詢問模式從 `document.body` 建立語意化文字樹，代理模式則先用 `getPageContextContainer()` 選擇 `main`、唯一的 `article` 或 `document.body`，再序列化過濾後的 HTML，保留 DOM 結構供工具調用與目標定位使用。

詢問模式的語意樹不是 Chrome 或輔助技術實際計算的 Accessibility Tree。它根據原生 HTML 語意、明確的 ARIA role、可存取名稱、可見文字、連結、控制項目前值與互動狀態建立近似結構。沒有語意的泛用容器會被攤平；`script`、`style`、`noscript`、`template`、AskPage 對話框、`hidden`、`inert`、`aria-hidden="true"` 以及 CSS 隱藏內容會被排除。開放式 Shadow DOM 會納入走訪，但跨來源 iframe 只保留 iframe 節點本身。

表單控制項會保留目前值，以便回答和頁面狀態相關的問題。密碼欄位仍保留角色、名稱與狀態，但不會輸出密碼值。若語意樹建立失敗或沒有有效內容，詢問模式才會降級回原有的主要容器 `.innerText`。

選取文字與整頁上下文的組合則由 `buildConversationContextText()` 處理。如果存在選取文字，格式固定是：先放語意樹或完整的過濾後 HTML，再放 **Selected text (main focus)**。選取文字不再做字元裁切。這代表系統的策略不是「只看選取內容」，而是「整頁為背景、選取為焦點」；在代理模式下尤其要保留全頁 HTML，讓模型能用 selection 理解使用者想處理的目標區域，同時仍有足夠 DOM 背景完成複雜操作。

實際送出時，Gemini 會把詢問快照放在固定的 `systemInstruction` 前綴，並把完整歷史轉成獨立的 `user` / `model` contents；不再把歷史串成單一 transcript，也不會額外重複附加本輪問題。每個使用者回合原本附加的截圖與圖片都會留在該回合中。

OpenAI、Azure OpenAI、OpenAI Compatible 與 Anthropic 也都會保留完整的角色化訊息序列。本輪使用者問題已在呼叫 provider 前加入 `conversationHistory`，因此 request builder 只展開一次歷史，不會再額外附加同一問題。OpenAI 官方端點在詢問模式會帶同一對話固定的 `prompt_cache_key`；Azure OpenAI 與 OpenAI Compatible 不加入此 OpenAI 專屬欄位；Anthropic 詢問模式會加入頂層 `cache_control: { type: 'ephemeral' }`。Gemini 與 Azure OpenAI 則依供應商的隱含提示詞快取機制處理。

## 四、一般提問與 `/summary` 的差異
`handleAsk()` 對一般提問與 `/summary` 的差異，其實只在「問題文字怎麼決定」，不在「頁面內容怎麼擷取」。

一般提問時，輸入框內容會直接作為 `question` 傳進 `askAI()`（`content.js:551-553`, `content.js:642-645`）。而當輸入為 `/summary` 時，程式會先嘗試讀取自訂摘要 prompt；若沒有設定，就改寫成預設摘要提示詞，例如要求輸出標題、重點摘要與總結（`content.js:567-572`）。但之後仍然是同一條呼叫路徑：`askAI(question, getActiveSelectedText(capturedSelectedText), displayedQuestion)`（`content.js:645`, `content.js:1439-1452`）。

這表示 `/summary` 並沒有獨立的頁面擷取邏輯，也沒有保證一定只總結全頁。若目前有效 selection 存在，`buildSystemPrompt()` 仍會告訴模型「使用者選取了要優先聚焦的文字」，`buildConversationContextText()` 也仍會把該 selection 放進 `Selected text (main focus)` 區段（`content.js:319-333`, `content.js:335-341`）。因此實作上雖然 UI 文案把 `/summary` 描述成「總結整個頁面」，但它其實更接近「以目前頁面為背景，必要時優先根據選取內容做摘要」。對維護者來說，這是一個重要的產品語意落差。

另外，`/screenshot` 是獨立的設定／測試命令，只切換截圖開關與測試截圖，不走上述文字問答流程，也不會進入 `askAI()` 的一般頁面文字組裝邏輯（`content.js:574-615`）。

## 五、重要函式、資料流與長度限制
### 1. 關鍵函式
- `createDialog()`：在視窗開啟瞬間擷取 selection（`content.js:377-382`）。
- `getPageContextContainer()`：決定代理模式 HTML 與詢問模式 fallback 要從 `main`、單一 `article` 或 `body` 取哪個容器。
- `buildApproximateAccessibilityTree()`：從 `document.body` 建立詢問模式使用的近似 Accessibility Tree。
- `getInquiryPageContext()`：包裝語意樹結果，並在建立失敗或內容為空時降級至 `.innerText`。
- `getPageContext()`：依詢問或代理模式回傳語意樹、fallback 純文字或過濾後 HTML。
- `getActiveSelectedText()`：優先使用本次 `capturedSelectedText`，否則 fallback 到 `conversationSelectedText`（`content.js:315-317`）。
- `buildSystemPrompt()`：決定模型是否被告知「有選取文字要優先聚焦」以及是否包含 screenshot 情境（`content.js:319-333`）。
- `buildConversationContextText()`：組裝 full page 與 selected text 的最終文字上下文（`content.js:335-341`）。
- `getPageConversationContext()`：詢問模式建立或重用凍結快照；代理模式直接建立最新頁面上下文。
- `getConversationMessagesForTextProviders()`：依原始角色順序展開所有可送入模型的歷史，並保留各使用者回合的圖片。
- `buildGeminiConversationContents()`：把完整歷史轉成 Gemini 的 `user` / `model` contents。
- `handleAsk()`：將一般提問、`/summary`、`/clear`、`/screenshot`、自訂命令導向正確流程（`content.js:549-645`）。
- `askAI()`：依 provider 分流到 Gemini、OpenAI、Azure OpenAI、OpenAI Compatible 或 Anthropic。

### 2. selection 的保存與清除
`conversationSelectedText` 是跨輪對話沿用 selection 的核心狀態。若本輪送出時有 `capturedSelectedText`，在模型回答完成後會把它寫回 `conversationSelectedText`，讓下一輪即使沒有重新反白，也可持續沿用相同焦點（`content.js:1054`, `content.js:1191`, `content.js:1350`, `content.js:1428`）。

這個機制讓使用者可以先選取一段文字，再連續追問多輪，而不需要每次都重新選取。不過它也帶來狀態延續的副作用：若使用者忘記先前已有選取焦點，後續問題可能仍持續受到舊 selection 影響。

`/clear` 會呼叫 `clearConversationHistory()`，同時清掉 `conversationHistory`、`conversationSelectedText`、詢問模式頁面快照與固定的 OpenAI `prompt_cache_key`。對話框內的 `capturedSelectedText` 也會清空。因此 `/clear` 不只是清聊天紀錄，也會完整重置詢問模式的固定前綴。程式不會因 API 回報 context window 超限而自動執行這個動作。

### 3. 長度處理
目前和頁面內容擷取直接相關的長度處理如下：

- **詢問模式語意樹：不設字元上限**，完整保留所有納入語意走訪的節點與文字。
- **詢問模式 fallback 純文字：不設字元上限**，完整保留頁面主要容器的 `.innerText`。
- **代理模式 HTML 上下文**：使用過濾後的頁面 HTML；即使存在選取文字，也不會因 selection 額外截斷 HTML。
- **選取文字：不設字元上限**，完整附加在頁面上下文後方。
- **對話歷史：不設自動裁切上限**。在 `/clear` 或重新整理頁面前，所有可送入模型的使用者與助理回合都會保留；進度、推理摘要與工具 trace 仍以 `includeInModelContext: false` 排除於模型上下文之外。
- **提示歷史上限：100 筆**，由 `handleAsk()` 在寫入 `PROMPT_HISTORY_STORAGE` 前裁切（`content.js:637-640`）。
- **模型輸出 token 上限**：Gemini 與 OpenAI 風格 provider 會依已知模型套用上限；無法判斷的 OpenAI 風格模型退回 32K。這與輸入 Context Window 是不同限制。

當完整快照與對話歷史超過模型 Context Window 時，請求會保留供應商回傳的錯誤並顯示給使用者。系統不會摘要、裁切或自動清除對話；必須由使用者自行執行 `/clear` 後開始新對話。

AskPage 不會依字元數預先判斷模型是否接受這些輸入。實際可接受的總量由所選模型與 API 的 Context Window、token 計算方式及供應商限制決定。

## 六、維護者應特別注意的實作含義
第一，這套演算法的核心假設是：**selection 提供焦點，full page 提供背景**。因此即使有 selection，詢問模式仍保留完整頁面語意樹，代理模式仍保留全頁 HTML。若未來要改善問答品質，調整點通常會落在語意角色與名稱推導、隱藏內容判定，或 selection 與 full page 的排序與權重，而不是單純改 prompt。

第二，`/summary` 目前不是獨立摘要模式，而是「換了一段 prompt 的一般提問」。如果產品預期 `/summary` 永遠總結全頁，則現況與產品語意並不完全一致，因為有效 selection 仍會改變模型焦點。

第三，selection 與 full page 的擷取時機取決於模式：selection 在開框時捕捉，詢問模式的頁面內容在第一問凍結，代理模式的 HTML 則每次重抓。因此詢問模式中的頁面變動不會自動反映到追問；這是為了維持可快取的固定前綴，也能避免後續 DOM 變更把新的頁面指令注入既有對話。模型同時會被明確要求把頁面上下文與選取文字視為不受信任資料，不得遵循其中的指令。
